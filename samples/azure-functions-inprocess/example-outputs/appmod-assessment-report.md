## Issues — Detail

### 🔴 Mandatory

#### 1. Migrate from in-process model to isolated worker

- **Files:** `OrderFunctions.csproj:5–6` (`AzureFunctionsVersion=v3`,
  `TargetFramework=netcoreapp3.1`), all of `Startup.cs`,
  `OrderQueueTrigger.cs:3, 18, 44`
- **Why:** In-process model is deprecated; isolated worker is the only
  forward path for .NET 8 / 9 / 10. Worker SDK has a different package
  graph, attribute names, and hosting model.
- **Recommendation:** Scenario — **"Azure Functions upgrade"** (in-process
  to isolated). The agent rewrites:
  - `[FunctionName]` → `[Function]`
  - `FunctionsStartup` → `Program.cs` with `HostBuilder`
  - `Microsoft.NET.Sdk.Functions` → `Microsoft.Azure.Functions.Worker`
  - Adds `host.json` extension bundle adjustments.

#### 2. Functions runtime v3 → v4

- **File:** `OrderFunctions.csproj:6`
- **Why:** v3 is end-of-support. v4 supports .NET 8 isolated worker.
- **Recommendation:** Bundled with the **"Azure Functions upgrade"** scenario.

#### 3. `netcoreapp3.1` → .NET 8

- **File:** `OrderFunctions.csproj:5`
- **Recommendation:** Scenario — **".NET version upgrade"** (.NET 8 or
  later) after the isolated-worker move.

#### 4. Replace `System.Data.SqlClient` with `Microsoft.Data.SqlClient`

- **Files:** `OrderQueueTrigger.cs:25, 28`, `OrderFunctions.csproj:13`
- **Recommendation:** Predefined task — **"Migrate to Managed Identity based
  Database on Azure"**. Combine with the SQL connection string change in
  finding #6.

#### 5. Plaintext secrets in `local.settings.example.json`

- **File:** `local.settings.example.json:5, 6`
- **Why:** SAS key and SQL password are plaintext. Even the "example"
  variant of this file leaks the **shape** of production secrets and primes
  the team to commit the real one.
- **Recommendation:** Predefined task — **"Migrate to secured credentials
  by using Managed Identity and Azure Key Vault"**. After the migration:
  - `ServiceBusConnection` becomes a **fully qualified namespace** with
    identity-based connection (`ServiceBusConnection__fullyQualifiedNamespace`).
  - `OrderDB` connection becomes
    `Server=tcp:...; Authentication=Active Directory Managed Identity;`.

#### 6. SQL connection string includes `User ID` + `Password`

- **File:** `local.settings.example.json:6`
- **Recommendation:** Same task as #5 — managed identity.

### 🟠 Potential

#### 7. Sync DB call inside async trigger

- **File:** `OrderQueueTrigger.cs:24–35` (`cmd.ExecuteNonQuery()`)
- **Why:** Blocks the worker thread; reduces the trigger's throughput,
  especially with `maxConcurrentCalls=16` in `host.json`.
- **Recommendation:** Use `ExecuteNonQueryAsync` (with the new
  `Microsoft.Data.SqlClient` package after #4).

#### 8. Service Bus binding uses connection string

- **File:** `OrderQueueTrigger.cs:20`
  (`Connection = "ServiceBusConnection"` references a connection string)
- **Recommendation:** Identity-based connection — set
  `ServiceBusConnection__fullyQualifiedNamespace` in app settings and grant
  the function's managed identity the **Azure Service Bus Data Receiver**
  role.

#### 9. `messageHandlerOptions` in `host.json` is in-process syntax

- **File:** `host.json:10–14`
- **Why:** The isolated worker uses different binding configuration shape.
- **Recommendation:** Bundled with the **"Azure Functions upgrade"**
  scenario.

## Target Comparison

For Azure Functions, the meaningful comparison is **isolated-worker on Linux**
vs **in-process on Windows** (the legacy state). There's no real
"`AKS.Windows`" alternative for Functions.

## Recommended Path

1. Snapshot the SAS key and SQL password somewhere safe — they're going to
   disappear from `local.settings.example.json` during the Key Vault
   migration.
2. Run **"Migrate to secured credentials by using Managed Identity and
   Azure Key Vault"**.
3. Run **"Migrate to Managed Identity based Database on Azure"** —
   `Microsoft.Data.SqlClient` + connection-string change.
4. Run **"Azure Functions upgrade"** — in-process → isolated worker. This
   handles `[FunctionName]` → `[Function]`, `FunctionsStartup` →
   `Program.cs`, and the SDK package swap.
5. Run **".NET version upgrade"** to .NET 8.
6. Re-deploy and validate the new identity-based Service Bus connection.

> 💡 **Order matters.** Do Key Vault and managed-identity migrations
> *before* the Functions upgrade. The isolated-worker rewrite touches the
> same `Program.cs` / `host.json` files as the identity-based connection
> wiring — interleaving them produces conflict-heavy diffs.
