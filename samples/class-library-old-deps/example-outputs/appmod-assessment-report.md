## Issues — Detail

### 🔴 Mandatory

#### 1. Upgrade `Newtonsoft.Json` (and remove `TypeNameHandling.Auto`)

- **File:** `JsonExtensions.cs:3, 12, 17–22`
- **Why:**
  - `Newtonsoft.Json 9.0.1` is six major versions behind. While `Newtonsoft`
    still works on .NET 8, the agent-driven path is `System.Text.Json`.
  - `TypeNameHandling.Auto` allows untrusted input to instantiate arbitrary
    types — a known **deserialization gadget** vector (`CWE-502`). Even if
    this code currently sees only trusted input, leaving it in is a footgun
    for the next consumer.
- **Recommendation:**
  - Scenario — **"Newtonsoft.Json upgrade"** (System.Text.Json migration).
  - Even if you keep Newtonsoft, **delete `TypeNameHandling.Auto`** and rework
    polymorphism with `JsonConverter<T>`.

#### 2. Replace `System.Net.Http 4.3.0`

- **File:** `Contoso.Mapping.csproj:25–27`
- **Why:** Multiple CVEs documented in the
  [advisory database](https://github.com/advisories?query=System.Net.Http+4.3.0).
  Removing this package and using the framework-supplied `HttpClient`
  (.NET 8) eliminates the issue.
- **Recommendation:** Drop the explicit `<Reference Include="System.Net.Http">`
  during the SDK-style conversion + .NET 8 upgrade — the framework ships it.

#### 3. AutoMapper 6.x → 12+ migration (static API removed)

- **File:** `MappingService.cs:3, 12–18, 22`
- **Why:** `Mapper.Initialize` and `Mapper.Instance` were removed in
  AutoMapper 9.0 (2019). The current sample API simply doesn't compile
  against modern AutoMapper.
- **Recommendation:**
  - Inject `IMapper` via DI.
  - Move profile registration to `MapperConfiguration` + `AddAutoMapper(typeof(MyProfile).Assembly)`.
  - This is **not** a predefined task — it's a manual refactor flagged in
    the report so consumers know it's coming.

#### 4. Convert `packages.config` to `<PackageReference>`

- **Files:** `packages.config`, `Contoso.Mapping.csproj:13–28`
- **Why:** SDK-style `csproj` and the `dotnet` CLI require `PackageReference`.
  The agent's other tasks expect SDK-style.
- **Recommendation:** Scenario — **"SDK-style conversion"**.

#### 5. Upgrade from .NET Framework 4.5.2

- **File:** `Contoso.Mapping.csproj:11`
- **Recommendation:** Scenario — **".NET version upgrade"** (`netstandard2.0`
  if the library has many .NET Framework consumers, or `net8.0` if all
  consumers can move).

### 🟠 Potential

#### 6. log4net direct usage

- **File:** `LoggingHelper.cs:1–17`
- **Why:** log4net itself runs on .NET 8, but exposing `LoggingHelper.Info(...)`
  pulls log4net into every consumer. The cloud-native pattern is
  `Microsoft.Extensions.Logging` with the `ILogger<T>` injected by the
  consumer.
- **Recommendation:** Replace `LoggingHelper` with an injected
  `ILogger<MappingService>`. log4net can stay if a consumer requires it,
  via `Microsoft.Extensions.Logging.Log4Net.AspNetCore` adapter — but
  prefer to delete it.

#### 7. log4net `FileAppender` writes to `C:\Logs\...`

- **File:** `App.config:5`
- **Why:** Won't exist on Linux containers. Consumer apps won't surface a
  useful error if the path can't be created.
- **Recommendation:** Remove the appender after switching to OpenTelemetry
  in the consumer. If log4net stays, configure the path from environment.

#### 8. `HttpClient` static field in `JsonExtensions`

- **File:** `JsonExtensions.cs:9`
- **Why:** Static `HttpClient` is the recommended pattern, but in a library
  it caches DNS results indefinitely and survives `IConfiguration` reloads.
  `IHttpClientFactory` (injected by consumers) is the modern answer.
- **Recommendation:** Take `HttpClient` as a constructor parameter so
  consumers can wire it via `IHttpClientFactory`.

## Recommended Path

1. Run **"SDK-style conversion"** to move from `packages.config` to
   `PackageReference`. Side effect: `System.Net.Http 4.3.0` reference
   becomes deletable.
2. Run **"Newtonsoft.Json upgrade"** to migrate to `System.Text.Json` *and*
   remove `TypeNameHandling.Auto`.
3. Manually refactor `MappingService` to instance-based `IMapper` (plus a
   `MappingProfile`).
4. Manually refactor `LoggingHelper` to `ILogger<T>` injection. Remove
   `App.config` `<log4net>` section. Document the change in CHANGELOG —
   this is a breaking change for consumers.
5. Run **".NET version upgrade"** — choose `netstandard2.0` if you must
   support both .NET Framework and modern consumers; otherwise `net8.0`.
6. Re-publish the package, bump major version (this is a breaking-change
   release).

> 💡 **Class libraries are special.** Run the assessment on the library
> *and* on at least one consumer in the same wave. If the consumer's
> assessment changes after the library upgrade, the library upgrade isn't
> done yet.
