## Issues — Detail

### 🔴 Mandatory

#### 1. ASP.NET WebForms is not supported on .NET 8+

- **Files:** `Default.aspx` (entire markup), `Default.aspx.cs:11, 30, 65, 78`,
  `Global.asax.cs` (entire file)
- **Why it matters:** WebForms (`System.Web.UI.Page`, `<asp:*>` server
  controls, ViewState, postback events) only runs on .NET Framework. There is
  no compatibility pack.
- **Recommendation:**
  - If the team is **not ready to rewrite**: target `AppService.Windows` or
    `AKS.Windows` and treat this as a lift-and-shift.
  - If the team **is ready**: rewrite as **Blazor Server** (closest paradigm
    to WebForms — server-side rendering, postback-like interactivity) or
    **ASP.NET Core MVC / Razor Pages**.

#### 2. `System.Web.HttpContext` usage

- **Files:** `Global.asax.cs:14, 21`
- **Why it matters:** Doesn't exist in ASP.NET Core. Replace with
  `IHttpContextAccessor`.

#### 3. `<sessionState mode="InProc">`

- **File:** `Web.config:14`
- **Why it matters:** In-process session state pins users to a single
  instance — breaks immediately on scale-out (App Service multiple instances,
  ACA replicas, AKS pods).
- **Recommendation:** Predefined task — **"Migrate to Azure Cache for Redis
  by using Managed Identity"** (use as distributed session store).

#### 4. SQL injection in `BindEmployees`

- **File:** `Default.aspx.cs:43–44` (`department` concatenated into `SELECT`)
- **Why it matters:** Live vulnerability.
- **Recommendation:** Parameterize — `WHERE Department = @dept`.

#### 5. Replace `System.Data.SqlClient` and `Integrated Security=True`

- **Files:** `Default.aspx.cs:5, 38`, `Web.config:4–6`
- **Recommendation:** Predefined task — **"Migrate to Managed Identity based
  Database on Azure"**.

#### 6. Replace `System.Diagnostics.EventLog` in `Application_Error`

- **File:** `Global.asax.cs:18–22`
- **Recommendation:** Predefined task — **"Migrate to OpenTelemetry on
  Azure"**.

#### 7. Upgrade from .NET Framework 4.5

- **File:** `EmployeePortal.csproj:11`
- **Recommendation:** Scenario — **".NET version upgrade"** (.NET 8 or
  later).

### 🟠 Potential

#### 8. Forms authentication (`<authentication mode="Forms">`)

- **File:** `Web.config:13`
- **Why it matters:** Forms auth still works on `AppService.Windows`, but it's
  legacy and not recommended for new development. Modern equivalent is
  ASP.NET Core Identity or **Microsoft Entra ID**.
- **Recommendation:** Predefined task — **"Migrate to Microsoft Entra ID"**.

#### 9. ViewState

- **Files:** `Default.aspx` (every `runat="server"` control),
  `Web.config:15` (`enableViewState="true"`)
- **Why it matters:** Increases payload size and is a known data-leak vector
  if not encrypted. ASP.NET Core has no ViewState.
- **Recommendation:** Re-architect state as part of the Blazor / MVC rewrite.

#### 10. `compilation debug="true"` left enabled

- **File:** `Web.config:11`
- **Why it matters:** Disables compiler optimizations; should be `false` in
  production.
- **Recommendation:** Set to `false`.

## Recommended Path

The honest decision is **rewrite vs. lift-and-shift**:

### If lifting

1. Fix the SQL injection.
2. Run **"Migrate to Microsoft Entra ID"** to replace Forms auth.
3. Run **"Migrate to Azure Cache for Redis by using Managed Identity"** for
   distributed session state (so you can scale beyond 1 instance).
4. Run **"Migrate to secured credentials by using Managed Identity and Azure
   Key Vault"** + **"Migrate to Managed Identity based Database on Azure"**.
5. Deploy to `AppService.Windows`.

### If rewriting

1. Plan a **Blazor Server** rewrite (closest paradigm to WebForms).
2. Reuse the data access layer after fixing the SQL injection and migrating
   to managed identity.
3. Replace `Page_Load` + postback events with Blazor lifecycle methods and
   event handlers.
4. Replace `<asp:GridView>` with Blazor `QuickGrid` or `MudBlazor` data
   grids.
5. Replace ViewState with explicit `[Parameter]` properties and component
   state.
6. Deploy to `AppService.Linux` or `ACA`.
