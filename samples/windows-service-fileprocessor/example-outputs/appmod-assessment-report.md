## Issues — Detail

### 🔴 Mandatory

#### 1. Replace MSMQ (`System.Messaging`) with Azure Service Bus

- **File:** `MessageQueueListener.cs:5, 14, 19, 24, 27, 41–58`
- **Why it matters:** `System.Messaging` is .NET Framework only and was
  removed from .NET Core / .NET 5+. There is no shim. MSMQ also has no
  managed Azure-native equivalent — the right replacement is **Azure Service
  Bus** with a `ServiceBusProcessor`.
- **Recommendation:** Predefined task — **"Migrate to Azure Service Bus"**.

#### 2. Replace `ServiceBase` hosting with `IHost` Worker Service

- **Files:** `Program.cs:7`, `FileProcessorService.cs:11, 22, 28–48`
- **Why it matters:** `System.ServiceProcess.ServiceBase` runs only on
  Windows under SCM. The container-friendly equivalent is
  `Microsoft.Extensions.Hosting` with one or more `BackgroundService`
  classes. Same lifecycle (`StartAsync` / `StopAsync`), much better cancellation
  + DI + logging.
- **Recommendation:** Scenario — **".NET version upgrade"**, then convert to
  `BackgroundService`.

#### 3. Replace `System.Diagnostics.EventLog`

- **Files:** `FileProcessorService.cs:30, 47, 67`,
  `MessageQueueListener.cs:54, 60`
- **Why it matters:** Windows-only API.
- **Recommendation:** Predefined task — **"Migrate to OpenTelemetry on
  Azure"**; use `ILogger<FileProcessorService>` with the Application Insights
  exporter.

#### 4. Upgrade from .NET Framework 4.5

- **File:** `FileProcessor.csproj:9`
- **Recommendation:** Scenario — **".NET version upgrade"** (.NET 8 or later).

#### 5. Local filesystem paths (`C:\Inbox\...`)

- **Files:** `App.config:4–5`, `FileProcessorService.cs:13–14, 31, 53–60`
- **Why it matters:** Containers don't have `C:\Inbox`. Paths must be either
  configurable env vars *and* point to mounted volumes, or replaced with cloud
  storage entirely.
- **Recommendation:** Predefined task — **"Migrate to Azure Blob Storage"**
  (recommended) or **"Migrate to Azure File Storage"** if a real filesystem
  is required.

### 🟠 Potential

#### 6. Synchronous I/O on timer thread

- **File:** `FileProcessorService.cs:53–60, 73–77`
- **Recommendation:** Convert to `async` end-to-end inside a
  `BackgroundService.ExecuteAsync` loop.

#### 7. No cancellation token plumbing

- **Files:** `FileProcessorService.cs:30, 53, 73`,
  `MessageQueueListener.cs:30, 41`
- **Why it matters:** The current shutdown relies on `volatile bool _running`
  + `Thread.Sleep`. `BackgroundService` and Service Bus processors expect
  `CancellationToken` propagation for clean shutdown.
- **Recommendation:** Adopt `CancellationToken` everywhere during the rewrite.

#### 8. `XmlMessageFormatter` with `typeof(string)`

- **File:** `MessageQueueListener.cs:23`
- **Why it matters:** Implicit serialization contract; brittle on the receiver
  side. Service Bus uses byte payloads — pick `System.Text.Json` and a
  versioned schema.

#### 9. No retry / dead-letter handling on MSMQ failures

- **File:** `MessageQueueListener.cs:55–63`
- **Recommendation:** Service Bus processor handles dead-lettering natively;
  configure `MaxDeliveryCount` and a DLQ handler.

**Reading the table:**

- `AKS.Windows` is the *lift-and-shift* path — costs nothing in code change
  but inherits the entire Windows-node operating cost and a dead-end runtime.
- `ACA` is the **modernize-and-run** path — every Mandatory item maps to a
  predefined task or a small refactor, and the result scales to zero.
- `AKS.Linux` matches `ACA` work-wise but adds Kubernetes operating overhead.

## Recommended Path

1. **Branch.** Create `modernize/file-processor` and lock it down — no merges
   to `main` until validation.
2. Run **"Migrate to Azure Service Bus"** — replace MSMQ first, since the rest
   depends on the new message contract.
3. Run **"Migrate to OpenTelemetry on Azure"** to replace EventLog.
4. Run **"Migrate to Azure Blob Storage"** to replace `C:\Inbox\...`.
5. Convert `ServiceBase` to `BackgroundService` as part of the
   **".NET version upgrade"** scenario (.NET 8).
6. Containerize and deploy to **Azure Container Apps**.
