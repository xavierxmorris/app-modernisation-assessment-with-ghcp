## Issues — Detail

### 🔴 Mandatory

#### 1. Upgrade from .NET Framework 4.5 to .NET 8

- **File:** `OrderClient.csproj:9`
- **Why:** WPF is fully supported on .NET 6 / 8 / 10. .NET Framework 4.5
  is end of life.
- **Recommendation:** Scenario — **".NET version upgrade"** (target `net8.0-windows`).
  - The agent rewrites `<TargetFrameworkVersion>v4.5</TargetFrameworkVersion>` to
    `<TargetFramework>net8.0-windows</TargetFramework>`.
  - Adds `<UseWPF>true</UseWPF>` (already present in this sample).
  - Removes explicit `PresentationCore` / `PresentationFramework` /
    `WindowsBase` references — they come from the SDK.

#### 2. Convert to SDK-style `csproj`

- **File:** `OrderClient.csproj` (entire)
- **Recommendation:** Scenario — **"SDK-style conversion"**.

#### 3. Plaintext `ApiKey` in `App.config`

- **File:** `App.config:5`
- **Why:** Desktop apps that ship config with embedded API keys leak the
  key to **every user** — anyone with `notepad App.config` access can read
  it. Azure Key Vault doesn't apply to per-user desktop installs the same
  way it does for server-side code, so the right answer differs.
- **Recommendation:**
  - **Best:** delegate auth to **Microsoft Entra ID** with MSAL — the
    desktop app prompts for sign-in and gets a token; the API validates the
    token. No static key.
  - **OK fallback:** store the key in Windows **Credential Manager** /
    DPAPI per-user. Agent doesn't auto-do this — manual.
  - **❌ Avoid:** keeping a static key and obfuscating it. Obfuscation is not
    encryption.

#### 4. Replace `System.Diagnostics.EventLog`

- **File:** `OrderViewModel.cs:46–48`
- **Why:** Even desktop apps benefit from structured logging shipped to
  Application Insights — gives you crash telemetry across users.
- **Recommendation:** Predefined task — **"Migrate to OpenTelemetry on
  Azure"**. The Application Insights SDK has a desktop SKU that supports
  per-user installation.

### 🟠 Potential

#### 5. Synchronous `HttpWebRequest` on the UI thread

- **File:** `OrderViewModel.cs:34–43`, called from `MainWindow.xaml.cs:14`
  via `RefreshButton_Click`
- **Why:** Button click handler is on the UI thread. `req.GetResponse()`
  blocks until the HTTP call completes. Window freezes; users get the
  "(Not Responding)" title bar. Severity escalates if the API is slow.
- **Recommendation:**
  - Replace `HttpWebRequest` with `HttpClient` (modern, `IHttpClientFactory`
    if you wire DI).
  - Make `LoadOrdersSync` → `LoadOrdersAsync` (`async Task`).
  - Make `RefreshButton_Click` `async` and `await` the call.
  - The .NET version upgrade scenario suggests this rewrite mechanically.

#### 6. Hardcoded internal-only API URL

- **File:** `App.config:4` (`http://internal-orders.contoso.local/api`)
- **Why:** Won't work for users on home / customer networks. Desktop apps
  need either VPN, an internet-reachable API, or both.
- **Recommendation:** Move the URL to a per-environment config (the user's
  organisation publishes the right URL during install). If the API is
  internal-only by design, document the network requirement.

#### 7. Static `App.ApiBaseUrl` reads `ConfigurationManager.AppSettings`

- **File:** `App.xaml.cs:9–14`
- **Why:** Static state is hostile to unit testing, and `ConfigurationManager`
  is the .NET Framework idiom. Modern WPF often uses `IHost` + DI.
- **Recommendation:** During the .NET 8 upgrade, switch to
  `Microsoft.Extensions.Hosting` and inject `IConfiguration` /
  `IOptions<AppOptions>` into the view model. The sample is small enough
  that this is mostly mechanical.

## Recommended Path

1. Run **"SDK-style conversion"** to modernize `csproj`.
2. Run **".NET version upgrade"** (target `net8.0-windows`). This brings:
   - WPF on .NET 8.
   - `HttpClient` instead of `HttpWebRequest` (mechanical refactor).
   - Async-ifies the click handler.
3. Replace static `ApiKey` with **Microsoft Entra ID** sign-in (MSAL).
   This is **not** a predefined task — manual but high-leverage.
4. Add Application Insights via **"Migrate to OpenTelemetry on Azure"** for
   crash + usage telemetry.
5. Switch installer to MSIX (.NET 8 packaging) for clean per-user install.

> 💡 **Desktop modernization is mostly cheap.** Unlike WebForms or WCF, WPF
> survives the .NET version upgrade with minimal changes. The agent typically
> ships the upgrade in one PR per project — and the same wave can update
> dozens of small WPF utilities across the org.
